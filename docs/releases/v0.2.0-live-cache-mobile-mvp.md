# v0.2.0 — Live Cache Mobile MVP

Status: stable product-quality MVP milestone.

## Summary

This release turns B20 Watcher from a mock/live prototype into a usable live cached product demo.

The app now supports a production-oriented flow:

1. Discover B20 tokens through CDP SQL.
2. Refresh token timelines and risk reports into Supabase.
3. Serve public reads from Supabase cache.
4. Display a mobile-first dashboard and readable token reports.
5. Explain why each token receives its risk score.

Reference deployment:

```txt
https://b20-watcher.netlify.app
```

## Product state

B20 Watcher answers:

> Who controls this B20 token, and what issuer-control risks exist before I trust it?

This release focuses on issuer-control and operational risk. It does not predict price and is not investment advice.

## Major capabilities

### Live cached data

- Supabase cached mode is supported through `DATA_SOURCE=supabase`.
- Token snapshots, event timelines, and risk reports are stored in Supabase.
- Rows are scoped by network, so `base` and `base_sepolia` can coexist.
- Normal page reads are pure reads and do not refresh or write to the cache.

### Refresh layer

- `POST /api/refresh/recent` refreshes recent B20 tokens.
- `POST /api/refresh/token/[address]` refreshes one token.
- Refresh is protected by `x-refresh-secret`.
- GitHub Actions scheduled refresh is available through `.github/workflows/refresh-cache.yml`.
- Manual workflow runs can pass a small limit such as `5` for safe smoke tests.

### Diagnostics

- `/api/health` returns secret-free service status.
- `/api/diagnostics` checks CDP and Supabase connectivity/configuration without exposing secrets.
- `/api/diagnostics/cdp-probe` helps debug zero-row CDP discovery results.

### Mobile dashboard

- Dashboard has mobile-first token cards.
- Each mobile card shows:
  - token name and symbol;
  - copyable token address;
  - score;
  - event count;
  - main score reason;
  - generated timestamp;
  - `View report` link.
- Mobile cards have clear horizontal separators.
- Desktop still uses the full table layout.

### Search

Dashboard search supports:

- token name;
- symbol;
- address;
- risk level;
- score;
- score reason;
- risk flag title.

### Copyable values

The UI shortens and supports copying:

- token addresses;
- transaction hashes;
- role-holder addresses;
- relevant event arguments.

### Score explanation

The dashboard and token report now explain scores.

Example:

```txt
Score: 10/100
Main reason: Supply cap changed +10
```

Token reports include:

- main reason panel;
- score breakdown card;
- score-bearing flags with points;
- note that score is the capped sum of score-bearing flags.

### Event timeline quality

Raw event names are translated into clearer labels:

```txt
SupplyCapUpdated → Supply cap updated
RoleGranted      → Role granted
PolicyUpdated    → Transfer policy updated
```

Timeline entries include short explanations, prioritized args, formatted large numbers, and copyable address/hash values.

Unreadable role bytes are displayed as `Unknown role` instead of broken unicode text.

## Verification checklist

After deployment:

```txt
/                         dashboard loads
/api/health               returns ok health payload
/api/tokens               returns cached token list
/tokens/<token-address>   renders token report
/api/risk/<token-address> returns full report JSON
```

Dashboard checks:

- search filters tokens;
- mobile card layout appears on phone;
- no horizontal table scroll on mobile;
- card separators are visible;
- score, events, and main reason are visible;
- copy button copies full address;
- `View report` opens token detail page.

Token report checks:

- token address is copyable;
- score and level are visible;
- main reason explains the score;
- score breakdown lists point-bearing flags;
- active roles section renders without overflow;
- timeline uses readable event titles and descriptions;
- transaction hashes and addresses are shortened and copyable;
- unreadable role values appear as `Unknown role`.

Refresh checks:

```bash
curl -X POST https://<your-domain>/api/refresh/recent \
  -H "Content-Type: application/json" \
  -H "x-refresh-secret: <REFRESH_SECRET>" \
  -d '{"limit": 5}'
```

Expected response shape:

```json
{
  "network": "base_sepolia",
  "tokens": 5,
  "events": 34,
  "reports": 5
}
```

Counts vary depending on network and live data.

## Security posture

- `CDP_BEARER_TOKEN` is server-side only.
- `SUPABASE_SERVICE_ROLE_KEY` is server-side only.
- `REFRESH_SECRET` protects refresh and diagnostics endpoints.
- Public read routes do not expose secrets.
- `/api/health` and diagnostics are designed to be secret-free.

## Known limitations

- Refresh is explicit; page reads do not auto-refresh.
- No event diffing between refresh runs yet.
- No alerts yet.
- Role decoding is partial.
- Unknown or unreadable roles are hidden in UI and ignored by scoring rather than crashing the app.
- Score reflects observed issuer-control capability, not intent.
- No x402 or paid API layer yet.
